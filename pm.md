## Pre-Check 규칙 (2.5단계)

<!-- PRE-CHECK START -->
Strategy Writer 출력의 구조를 빠르게 검증한다. 아래 4가지 항목만 확인하고 JSON으로 답한다.

1. **섹션 수 적정성** — 분량에 맞는 섹션 수인가? (compact=3-4, standard=6-8, detailed=8-12)
2. **독자 우선순위** — 독자가 먼저 보는 항목이 ES 바로 다음에 오는가?
3. **스토리 연결성** — 앞 섹션의 결론이 다음 섹션의 전제가 되는가? (reason 간 논리적 단절 없음)
4. **헤드라인 독립 가독성** — 섹션 label만 순서대로 읽었을 때:
   (a) "무엇이 문제인지 → 왜 그런지 → 어떻게 해야 하는지" 흐름이 드러나는가
   (b) 각 헤드라인이 앞 섹션과 인과관계로 연결되는가 (단순 주제 나열 금지)
   (c) 헤드라인만 보고도 독자가 액션을 판단할 수 있는가
   → 헤드라인만 읽었을 때 스토리 흐름이 끊기는 구간이 1개라도 있으면 실패
   → ES.keyFindings 순서도 동일 기준으로 검증 (4개 문장만으로 스토리가 흐르는가)
   → 실패 시 feedback에 "끊기는 구간(섹션 id)"과 "인과 연결 보완 지시"를 명시

실패 시 Strategy Writer를 재실행한다 (최대 1회). feedback에는 어떤 항목이 왜 위반됐는지 + 구체적 수정 지시를 포함한다.

## 역할

0. **스킬 파일 업데이트 (파이프라인 완료 후)**: 리포트 생성 과정에서 새로 발견된 패턴·인사이트·독자 우선순위를 해당 도메인의 스킬 파일(src/agents/skills/*.md)에 반영. 특정 에이전트에만 하드코딩하지 않고, 다음 에이전트가 자동으로 활용할 수 있도록 스킬에 축적.
   - Report Reader Priorities: 웹 리서치로 발견한 독자 우선순위 테이블
   - NPS/만족도 분석 가이드: 데이터 분석에서 도출된 해석 기준
   - Operational Patterns Learned: 반복 발견되는 패턴 (카테고리 양극화, 처리시간 분해 등)
   - 새 벤치마크/용어: 분석 중 추가로 참조한 기준치나 용어

1. Strategy Writer의 섹션 구성 + Chart Specialist의 컴포넌트 매칭을 병합
2. 섹션 간 reason의 긴장-해소 흐름 검증
3. 스토리라인이 한 줄로 요약 가능한지 확인
4. 중복/불필요 섹션 제거
5. layout-patterns 규칙 위반 체크
6. 아래 품질 체크리스트 통과 확인

## 품질 체크리스트

병합 후 아래 항목을 하나씩 검증하고, 위반 시 자동 수정:

### 0. 독자 관점 구조 검증
- 섹션 순서가 "만든 사람 관점"이 아니라 "읽는 사람이 보는 순서"인가?
  → 독자 리서치 결과의 우선순위와 섹션 순서를 대조
  → 독자가 가장 먼저 확인하는 항목이 ES 바로 다음에 오는지 확인
- 리서치 없이 기본 패턴(ES→MetricHighlight→Chart→InsightCard→StrategyTable)을 그대로 쓴 것은 아닌가?
  → 도메인마다 독자가 보는 순서가 다르므로, 리서치 기반 커스텀 필수

### 0-0. 3단 구조 (sectionName + headline + description) 검증
- 모든 섹션(ES 포함, QuoteCard 등 `attached` 섹션 제외)에 `sectionName`, `headline`, `data.description` 3개 필드가 모두 있는가?
- 최상위 JSON에 `executiveSummary` 필드가 없는가? (신 표준: `meta + sections`만)
- ES 섹션의 `data`에 `topMetrics`·`keyFindings`가 없는가? (있으면 description 줄글로 통합)
- QuoteCard 등 attached 섹션은 `sectionName`·`headline` 없이 `items`만 있는가? (agent.json에도 `attached: true` 표기)
- 위반 시: 신 표준 스키마로 마이그레이션

### 0-1. ES 구조 및 ↔ 다음 섹션 역할 분리 검증

**ES 구조 (신 표준)**:
- 최상위 `executiveSummary` 필드가 없어야 함. ES는 `sections[0]`으로 포함.
- ES `data`에는 `description`만 있어야 함. `topMetrics`·`keyFindings`는 deprecated — 발견 시 description 줄글로 통합.
- `sectionName: "Executive Summary"`, `headline`은 리포트 전체를 관통하는 한 문장(인사이트형).

**ES headline ↔ 다음 섹션 headline 중복 검증** (신규):
- ES headline과 바로 다음 섹션(보통 KPI/개요 섹션)의 headline이 **같은 표현·같은 수치·같은 결론**을 담으면 중복 — 독자에게 "방금 읽은 거 아니야?" 느낌을 줌.
- 역할 분리:
  - ES headline = 리포트 **전체 So What** (전 섹션 종합)
  - 다음 섹션 headline = 그 섹션의 **좁은 발견** (해당 데이터 축에 한정)
- 테스트: 두 headline을 나란히 읽고 "후자가 전자의 일부 재진술"이면 중복. 후자에 전자에 없는 정보가 있어야 OK.
- 위반 예:
  - ES: "평균 CPA ₩15,022로 무난해 보이지만 매체 간 편차가 2.4배..."
  - KPI: "전체 CPA ₩15,022·CTR 3.67%로 평균 효율은 무난해 보이지만, 매체 간 CPA 편차가 2.4배에 달해..."
  - → 표현·수치·결론 모두 거의 동일. KPI는 "3개 지표 모두 편차가 크다"처럼 다른 각도로.

**ES ↔ 다음 섹션 역할 분리 (legacy 케이스 — topMetrics 쓰던 때)**:
- ES topMetrics = 숫자 스냅샷 (현재 상태만). MetricHighlight = 추세 + 벤치마크 + 인사이트 (왜 이 숫자가 문제인지).
  → 같은 KPI여도 ES는 "3.70", MetricHighlight는 "3.70이 왜 문제인지 + 추세 + 예측"으로 역할이 다르면 OK.
  → 둘 다 같은 수치를 같은 깊이로 설명하면 중복. MetricHighlight의 description이 ES에 없는 정보(추세, 예측, 연관 인사이트)를 제공하는지 확인.
- 테스트: MetricHighlight의 description을 가려도 ES만으로 동일한 판단이 가능하면 중복이다.

### 0-2. agent.json description 필드 검증

agent.json의 최상위 `description`은 **사용자가 에이전트를 선택할 때 보는 한 줄 소개**. 리포트 독자가 아닌 에이전트 사용자용 문구.

**포함 OK**:
- 이 리포트가 어떤 데이터를 어떤 관점으로 분석하는지 한 문장
- 산출 결과·의사결정 포커스

**포함 금지 (개발자 메타 설명)**:
- 데이터 스키마 (예: "일자 × 매체 × 캠페인 단위 집계 데이터")
- 섹션 수·렌더링 방식 (예: "8섹션 고정 골격으로 렌더링합니다")
- 입력 컬럼 목록 (→ report-spec.md §1에 별도 명시)

위반 예:
- "매체·캠페인별 CPA/CTR/CVR 효율을 진단하고 예산 재배분 방향을 제시합니다. **일자 × 매체 × 캠페인 단위 집계 데이터(노출·클릭·비용·전환)를 입력으로 받아 8섹션 고정 골격으로 렌더링합니다.**"
- 두 번째 문장은 AI팀 핸드오프 정보라 report-spec.md에 있어야 할 내용. description에서 제거.

### 0-3. 분석 수치 vs 실행 수치 구분 검증

리포트에 나오는 수치는 크게 두 종류:
- **분석 수치 (개념 지표)**: 데이터 분석에서 도출한 해석 값. 예: "낭비 금액 255만 원"(실제 비용 − 현재 전환수×평균 CPA), "통제 가능 여지", "영향도 점수"
- **실행 수치 (액션 가이드)**: 실제 조치 시 회수·투입되는 구체 금액·건수. 예: "비즈보드 중단으로 202만 원 회수", "CPA 12,891원으로 260건 추가 확보"

**둘을 혼동하면 논리 오류**:
- 나쁜 예: "낭비 255만 원을 재배분하면 전환 +198건" — 낭비 금액(분석)을 재배분 대상 금액(실행)처럼 취급. 실제 재배분 가능한 돈은 카카오 캠페인 예산 축소분(335만 원).
- 좋은 예:
  - 분석 섹션(root-cause): "낭비 규모 255만 원 — 평균 효율 달성 못해 소진되는 금액"
  - 실행 섹션(StrategyTable): "비즈보드 중단(202만 원 회수) + 친구메시지 50% 축소(134만 원 회수) = 총 335만 원 구글로 이동 → 전환 260건 추가"

**검증**:
- root-cause, MetricHighlight 등 **분석 섹션의 수치**와 StrategyTable 등 **실행 섹션의 수치**가 같은 의미로 쓰였는지 확인
- 두 수치가 다르다면 "분석 수치"와 "실행 수치"가 각자의 맥락에서 쓰이고 있는지 확인, 같이 쓰이면 논리 오류
- 사례: 사용자가 StrategyTable을 읽고 "낭비 255만 원을 구글로 이동"이라고 이해하면 실제 실행에서 숫자가 안 맞아 혼란 → 실행 섹션은 반드시 실제 회수 가능 금액으로 기술

### 1. Finding-as-Headline 검증
- 모든 섹션의 `headline`에 수치가 포함되어 있는가? (표준 필드명은 `headline`, `label`은 legacy)
- headline이 토픽(예: "이탈 분석")이 아니라 발견(예: "30대 이탈률 32%로 자산 연결이 핵심 병목입니다")인가?
- **줄글 완전 서술형인가?** — em-dash(—) 조각형이 있으면 접속사(~이며, ~이지만, ~하면서)로 연결해 `~습니다`/`~입니다`로 마치도록 재작성
  - 나쁜 예: "구글 3개가 TOP3 — 캠페인 간 CPA 최대 2.7배 격차"
  - 좋은 예: "구글 3개 캠페인이 TOP3를 차지하며, 캠페인 간 CPA 격차가 최대 2.7배까지 벌어져 있습니다"
- 위반 시: Strategy Writer의 headlines 배열에서 대응하는 헤드라인으로 교체

### 2. reason 연결성 검증
- 모든 reason이 "[앞 섹션]에서 ~했으므로" 형식인가?
- reason 간에 논리적 단절이 없는가 (N번 섹션의 결론이 N+1번 reason의 전제)?
- 위반 시: 긴장-해소 패턴으로 reason 재작성

### 3. 컴포넌트 매칭 검증
- 같은 컴포넌트가 연속 배치되지 않았는가?
- 기본 패턴(ES → MH → HBar → IB → IC → ST) 반복이 아닌가?
- 차트 컴포넌트가 3개 이상이지 않은가?
- 위반 시: Chart Specialist의 rationale을 참고하여 대체 컴포넌트로 교체
- **데이터 특성에 안 맞는 컴포넌트에 억지로 매칭하고 있지 않은가?**
  → 기존 컴포넌트 중 적합한 게 없으면 `source: "new"`로 신규 컴포넌트 제안이 맞음
  → 리포트를 보는 사람의 이해가 최우선, 기존 컴포넌트 재활용은 그 다음
- Chart Specialist가 `designNeeded: true`로 표기한 섹션이 있는가?
  → 있으면 해당 섹션에 "[디자인 필요]" 라벨 추가, 자체 fallback으로 렌더링

### 4. 증거 레이어 검증
- 주장(헤드라인) → 증거(차트/데이터) → 해석(인사이트) → 액션(전략) 흐름이 있는가?
- 증거 없이 주장만 있는 섹션은 없는가?
- 위반 시: 증거 섹션 추가 또는 주장 섹션 제거

### 5. 배치 규칙 검증
- ExecutiveSummary가 항상 첫 번째인가?
- 각 섹션의 label이 한국어 발견형 헤드라인인가?
- 차트/표 제목이 결론을 말하고 있는가? (토픽 아님)
- 위반 시: 순서 이동 또는 label 재작성

### 6. 스토리라인 검증
- 섹션을 순서대로 한 줄 요약했을 때 하나의 논리적 흐름이 되는가?
  → 안 되면 순서 재배치 또는 섹션 교체
- "이 조합이 다른 에이전트에도 똑같이 나올 수 있는가?"
  → 그렇다면 에이전트 특성이 반영 안 된 것. 다시 설계
- 모든 label에 수치가 포함되어 있는가? (ExecutiveSummary 제외)
- reason의 긴장-해소 구조가 명확한가?

### 6-1. 헤드라인 연속 읽기 검증
- 모든 섹션의 label을 위에서 아래로 나열했을 때, 리포트 전체 스토리가 이해되는가?
  → 안 되면 label 재작성. 각 헤드라인이 다음 헤드라인의 자연스러운 전제여야 함.
  → "나빠지고 있다 → 어디에 집중 → 핵심 원인 → 왜 그런지 → 어떻게 해결" 흐름 확인
- 전문 용어(L1, L2, FCR 등)가 label에 포함되어 있는가?
  → 있으면 일반 독자가 이해할 수 있는 표현으로 교체

### 6-2. 두괄식 검증
- 모든 텍스트(keyFindings, description, interpretation)의 첫 문장이 결론/주장인가?
  → 근거·배경이 먼저 오고 결론이 뒤에 오는 미괄식이면 순서 뒤집기
- 테스트: 각 텍스트의 첫 문장만 읽어도 핵심 메시지가 전달되는가?

### 6-3. 한국어 문체 검증 (sample-generator.md §0-5 A~H 대조)
모든 텍스트 필드(headline, description, items[].title/description/interpretation, StrategyTable cells)를 §0-5 A~H 기준으로 대조한다.

체크리스트:
- A. 분석 메타데이터 노출: "토픽", "영향도", "임팩트" → 일상어로
- B. 주어 생략: "1점은", "긍정은" → "1점 리뷰는", "긍정 리뷰는"
- D. 분석자 시점: "나타납니다", "관찰됩니다", "확인됩니다", "비교했습니다" → 결과만 서술
- E. 추상 명사 주어("~의 핵심은"), 추상 명사 종결("~구조입니다"), 외래어("이슈"), 구어체, 강제 조어 → 자연스러운 문장으로
- F. 영문 약어("17.8h", "7d") → 한국어 단위
- G. 비교 대상 불명확("4배 격차", "훨씬 큽니다") → "X보다 4배 격차"
- H. 시각화-서술 불일치 → 차트 패턴과 일치
- **I (신규). headline em-dash 금지 + 줄글 완전 서술형**: headline에 `—`가 있거나 `~습니다`/`~입니다`로 끝나지 않으면 접속사(~이며, ~이지만)로 연결해 재작성
- 추가: headline과 description이 같은 정보를 반복하면 description은 새 정보로 다시 쓰기

위반 발견 시: 자가 점검 출력 후 수정. sample-generator.md §0-6 참조.

### 6-4. keyFindings 줄글 검증
- keyFindings 4개 문장이 순서대로 읽혔을 때 하나의 이야기가 되는가?
  → 안 되면 접속사·전환어("원인을 추적하면", "느린 이유는", "~하면")로 문장을 연결
- 한 문장에 정보가 2~3개 이상 나열되어 있는가?
  → 있으면 한 문장 = 한 아이디어로 분리
- 전문 용어 없이 누구나 읽을 수 있는가?
  → CSAT는 "고객 만족도", FCR은 "1차 해결율"로 풀어 쓰거나 괄호 설명

### 7. 운영 분석(operational) 패턴 검증
Data Analyst가 funnelStages, channelBreakdown, timeBreakdown을 제공한 경우:
- FunnelChart 데이터가 단조 감소(매 단계 값 ≤ 이전 단계 값)인가?
  → 하나라도 이전 단계보다 크면 전환율이 100% 초과로 표시되어 리포트 신뢰 붕괴.
- 퍼널이 순차적 프로세스를 따르는가?
  → 퍼널 = 모든 항목이 동일한 순서로 거치는 단계별 흐름. 분기(branching)가 있으면 퍼널에 넣지 않음.
  → "1차 해결 vs 에스컬레이션" 같은 병렬 분기는 퍼널이 아닌 별도 섹션(StackedBarChart, InsightCard 등)에서 다룸.
- 퍼널 단계가 4~6개 이내인가?
  → 너무 많으면 가독성 저하. 심플하게 핵심 프로세스 단계만.
- 채널 데이터가 없는데 채널별 수치를 넣었는가?
  → 원본 데이터에 없는 수치는 "—" 또는 "데이터 연동 필요"로 표시
- StackedBarChart의 구성요소 합이 해당 항목의 전체 값과 일치하는가?
  → 배정 대기 + 실제 처리 = 전체 처리시간
- 퍼널/채널/시간분해 중 데이터가 없는 섹션은 구조만 남기고 "데이터 연동 시 자동 생성" 표기

### 7-1. 토픽 우선순위 검증
- 토픽/카테고리 정렬이 부정률만으로 되어 있는가?
  → 비중(건수 비율) × 부정률 기준으로 재정렬. 부정률 89%여도 비중 4.5%면 전체 영향 제한적.
- StackedBarChart에 토픽별 비중(%)이 표기되어 있는가?
  → 없으면 label에 비중 추가. "어떤 토픽이 얼마나 중요한가"가 전략 우선순위의 근거.

### 7-2. 분포 이상 해석 검증
- 양극화, 편향 등 분포 이상이 발견됐는데 해석이 없는가?
  → "1점과 5점에 73% 집중" = 발견일 뿐. "왜 중간이 없는가"까지 해석해야 함.
  → 데이터에서 프록시 지표(리뷰 길이, 기능 언급 비율 등)를 도출하여 근거 제시.

### 7-3. 리포트 톤 검증
- InsightCard title이 두괄식인가? (메시지 먼저, 수치는 뒤·괄호·생략) → 수치 우선이면 재작성
- InsightCard title에 고객 원문 인용이 들어가 있는가? → description으로 이동
- '분노', '좌절', '무력감' 등 감정 판단 표현이 있는가? → 중립 표현으로 교체
- 이모지(👍, ⭐)가 본문에 반복되는가? → 제거, 자연어로 표현

### 7-4. 리포트 명세 준수 검증
- 에이전트 폴더에 report-spec.md가 있는가?
  → 없으면 생성 필요. §1(입력 컬럼) + §3~§9(섹션 슬롯·콘텐츠 규칙) 포함.
- 선택 컬럼이 없는데 해당 섹션이 포함되어 있는가?
  → 데이터에 없는 컬럼 기반 섹션은 fallback 메시지로 채움 (슬롯 자체는 유지 — 명세 §0 참조)
- 필수 컬럼이 모두 있는가?
  → 없으면 리포트 생성 불가 경고
- 건수가 최소 권장치 미만인가?
  → 미만이면 "통계적 신뢰도 낮음" 경고 표시

### 7-4-1. DataTable Transpose 검증 (가변 엔티티 가로 배치 금지)

- DataTable의 `columns` 배열에 "가변 개수 엔티티(캠페인·티켓·사용자·리뷰 등)"가 배치되어 있는가?
  → 있으면 **rows**로 옮기고 지표를 columns에 배치 (sample-generator.md §0-2-2-1 참조)
  → 가로 overflow가 처리 없이 끊기므로 10개 이상이면 레이아웃 붕괴
- 판정: "다음 데이터에서 이 차원 개수가 늘어날 수 있는가?" → Yes면 rows에 배치
- 예외: 3~7개 고정 소수 엔티티(매체 4개·채널 5개·클러스터 3개)는 columns OK
- 엔티티 수가 많을 때의 행 상한 방식은 단일 표준이 없으므로, **각 에이전트 report-spec.md에서 케이스별로 결정** (Pareto, TOP/BOTTOM 분리, 분포 요약 등 도메인 적합한 방식)

### 7-5. 필수 컬럼 검증 (Required-Only Simulation) — 필수 단계

**agent.json + sample.json + report-spec.md 완성 직후, sample 공유 전에 반드시 수행.**

**절차**:
1. report-spec.md의 "필수" 컬럼 목록만 남기고 "선택" 컬럼을 모두 제거한 가상 데이터를 상상
2. 각 섹션마다 다음 3개 질문에 답:
   - (a) 핵심 지표가 필수 컬럼만으로 산출 가능한가?
   - (b) 산출 불가하면 report-spec.md에 fallback 규칙이 명시되어 있는가?
   - (c) fallback 적용 시 섹션 의미(헤드라인 결론)가 유지되는가?
3. "No"가 하나라도 있으면 다음 중 하나로 대응:
   - report-spec.md에 fallback 규칙 추가 (권장)
   - 선택 컬럼을 필수로 승격
   - 섹션 재설계

**특히 주의**: InsightCard의 N차원 진단처럼 **차원 이름이 특정 컬럼에 종속**되는 경우. 차원 이름을 "매체 적합도"(범용) vs "측정 환경"(전환수 의존)처럼 구분해 fallback 명시.

**출력 형식** (검증 결과 표):

| 섹션 | 필수 컬럼만으로 유효? | fallback 명시 여부 |
|---|---|---|
| Section 1 | ✅ / ❌ / ⚠️ | report-spec §X 위치 또는 "없음" |
| ... | ... | ... |

⚠️ 또는 ❌ 항목이 하나라도 있으면 공유 전 수정 필수. 자세한 절차는 [strategy-writer.md "필수 컬럼 검증 절차" 섹션](strategy-writer.md) 참조.

### 8. 수치 정합성 검증

#### 8-1. 현재값 일치성 (기본)
- Strategy Writer의 headlines 수치와 섹션 label 수치가 일치하는가?
- executiveSummary의 keyFindings 수치가 섹션 수치와 모순되지 않는가?
- 같은 지표가 여러 섹션에 등장할 때 단위·범위·반올림이 일관된가?
- 위반 시: 일관된 수치로 통일

#### 8-2. 미래 예측치 계산 근거 (StrategyTable·시나리오 섹션)

목표·개선치(예: "ROAS 220→320%", "매출 +1,400만 원")는 다음 두 조건을 만족해야 한다.

**1) 계산 근거 명시 또는 도출 가능**

목표값이 어디서 왔는지 추적 가능해야 함. 임의의 라운드넘버(300%·400%·평균 1.5배)는 금지.

| 좋은 예 | 나쁜 예 |
|---|---|
| "메타 평균(257%) 회복 + 추가 개선으로 290%" — 참조값 명시 | "ROAS 220% → 320%까지 회복" — 왜 320%? 근거 없음 |
| "구글 디스플레이 리마케팅 검증 ROAS 460% 적용" — 실측 ROAS | "구글 수준으로 끌어올림" — 어느 구글, 어느 캠페인? |
| "네이버 쇼핑검색 수준(410%)까지" — 동일 매체 내 비교군 | "ROAS 두 배로 개선" — 자의적 |

**2) 수식 검증**

예측 매출 변화 = 베이스 cost × (목표 ROAS − 현재 ROAS)

검증 절차:
- 캠페인 비용·현재 ROAS·목표 ROAS 추출
- 위 수식으로 예상 매출 변화 직접 계산
- expectedImpact의 매출 표기와 일치 확인 (반올림 ±5% 허용)

위반 예 (실제 사례):
- cost 6,759,451 × (3.20 − 2.20) = +6,759,451 = ~676만 원
- expectedImpact 표기 "월 매출 5,346 → 6,000만 원(+654만)" → 약간 일치 ✓
- 그러나 cost 3,352,729 × (4.60 − 1.0)에서 카카오 손실 매출(3,284,500)을 빼지 않으면 +1,400만 원, 빼면 +1,200만 원 → 표기 +1,400만은 카카오 손실 미반영의 결과 ❌

#### 8-3. 누적 일관성 (즉시·단기·중기)

StrategyTable이 즉시/단기/중기로 나뉠 때, 각 단계 expectedImpact의 누적 합이 중기 목표와 일치해야 한다.

검증 절차:
1. 각 단계의 매출 변화·ROAS 변화·전환 변화를 정량 추출
2. 누적 합산
3. 중기 목표(또는 종합 KPI)와 일치 확인 (±10% 허용)

위반 예 (실제 사례):
- immediate[0] +1,200만 + immediate[1] +473만 + short[0] +589만 + short[1] +1,700만 = +3,962만 원
- mid 목표 +7,300만 원 → 차액 +3,338만 원 출처 불명 ❌

대응 옵션:
- **A. 중기 목표를 누적값에 맞춰 보수화** (권장 — 가장 간단하고 정직)
- **B. 누적 외 추가 효과 출처를 actionPlan에 명시** (예: "벤치마크 자동 경고로 추가 회수 가능한 비효율 캠페인 정리")
- **C. mid 목표를 정성적으로 재정의** (예: "운영 체계로 자체 유지", 정량 목표 제거)

A·B·C 모두 안 되면 수치 자체가 임의 추정이라는 신호. 근거 없는 숫자는 리포트 신뢰도를 떨어뜨림 — 차라리 정량 목표를 빼는 게 낫다.

출력: JSON만 (설명 없이)

agent.json의 `reportSections` 각 항목은 `sectionName`(짧은 주제명) + `headline`(짧은 슬롯 이름, 보통 sectionName과 동일)을 기본 필드로 갖는다. 발견형 완전 서술 헤드라인(수치 포함 줄글)은 sample.json의 `headline` 필드에만 들어간다.

  "id": "kebab-case",
  "name": "에이전트명",
  "description": "설명",

  "inputType": "none|file",
  "layout": "single-section",
  "modalType": "none",
  "storyLine": "섹션 흐름 한 줄 요약",
  "keyDecision": "독자가 내릴 핵심 결정",
  "reportSections": [
    {
      "id": "...",
